# 智能体多模态图文双擎记忆系统 (VGD Memory OS) 
## 需求与系统架构设计说明书 (V1.0)

### 1. 需求背景与系统建设目标

#### 1.1 业务与技术背景
在2026年的技术语境下，AI Agent正从“单次任务执行器”向“长生命周期数字生命”演进。传统的“超长上下文强塞”与“纯文本向量切片检索（Naive RAG）”在面临复杂物理与数字世界交互时，暴露出不可克服的瓶颈：**多跳逻辑断链、事实冲突导致的认知精神分裂、以及视觉时空记忆的严重缺失**。系统迫切需要重构底层记忆基础设施。

#### 1.2 系统建设目标
构建一套**视觉感知（Vision）、图脑逻辑（Gbrain）、长文叙事（Document）三位一体（VGD）**的高阶记忆操作系统：
1. **多模态物理对齐**：打破数据孤岛，实现图像节点、逻辑拓扑图谱与详尽维基页面的统一表征与联合推理。
2. **读写性能解耦（CQRS）**：彻底解决高阶记忆抽取导致的高延迟阻塞，实现“前台极速响应（快系统）”与“后台重度巩固（慢系统）”的双轨分离。
3. **100% 存储介质隔离**：定义高度标准化的持久化网关抽象接口，使系统支持零代码修改下的“热插拔”式底层存储更换。
4. **白盒化可视干预**：交付宏微观结合的“2.5D时空记忆宫殿”控制大盘，赋予人类对Agent“数字脑神经”的直观监测与外科手术级纠偏权限。

### 2. 核心业务场景与流程分析

系统架构基于“双过程认知理论”，核心业务流分为四大主干：

#### 2.1 极速工作记忆流 (System 1 - Fast Track)
* **场景**：Agent处理日常高频、低延迟交互。
* **流程**：多模态输入解析 $\rightarrow$ 提取向量特征压入轻量级内存/Redis滑动窗口 $\rightarrow$ 命中短期上下文后大模型极速响应 $\rightarrow$ 原始交互载荷异步抛入高吞吐消息队列（如Kafka），流程闭环，不阻塞主线程。

#### 2.2 异步深思与记忆巩固流 (System 2 - Slow Track)
* **场景**：系统空闲或Token积压达标后，后台重组长效记忆。
* **流程**：唤醒后台分布式 `Memory Worker` $\rightarrow$ 调度VLM提取图像的密集描述并生成图节点 $\rightarrow$ 调度LLM执行严谨的实体关系抽取（NER/RE） $\rightarrow$ 触发分布式Saga事务 $\rightarrow$ **[调用标准化存储网关]** 并发执行：构建拓扑边、覆写并消解Wiki文档冲突、落盘二进制图片。

#### 2.3 图文联合多跳推理流 (Hybrid VGD Retrieval)
* **场景**：处理跨时间、跨事件的极高复杂度侦探级任务。
* **流程**：多模态语义寻址（锁定起始节点UUID） $\rightarrow$ **[由图引路]** 在Gbrain中执行多跳游走，严格过滤噪音，锁定唯一正确的目标实体群UUID $\rightarrow$ **[由文生智]** 并行拉取目标UUID挂载的专属Wiki长文与历史图册 $\rightarrow$ 图文逻辑组装注入大模型生成严密回答。

#### 2.4 可视化白盒控制流 (Human-in-the-Loop)
* **场景**：开发者审计或用户手动重构AI记忆。
* **流程**：前端拉取宏观拓扑网与微观富文本 $\rightarrow$ 用户在3D视窗中“剪断”错误连线，或在富文本抽屉中篡改事实 $\rightarrow$ 触发逆向重构API，绕过AI反思机制，通过底层接口强制执行物理级状态重置。

### 3. 功能模块详细设计

#### 3.1 多模态感知与解析引擎 (Perception Engine)
* **对齐与降维器**：将不可压缩的图像像素流降维，不仅提取特征向量，更强制将其升格为图谱中的独立实体（`Image-as-a-Node`），并自动推演其与人物、时间的关系连线。
* **意图分流路由**：智能评估当前输入的长期价值，决定是否触发高成本的System 2重度巩固。

#### 3.2 记忆中枢调度编排器 (CQRS Orchestrator)
* **本体防漂移守护进程 (Anti-Ontology Drift)**：后台静默服务。定期运行社区发现与向量聚类算法，识别高相似度冗余节点（如“苹果手机”与“iPhone”），调用LLM裁判自动完成节点及Wiki页面的“无缝融合”。
* **Saga 状态机**：管理跨异构数据库的最终一致性。遇到局部介质（如文档库）宕机时，挂起事务并入死信队列（DLQ），恢复后执行补偿重试，杜绝“图文撕裂”。

#### 3.3 2.5D 可视化控制工作台 (Memory Dashboard)
* **宏观 3D 星图拓扑 (Macro Star-Graph)**：基于力导向物理引擎的 WebGL 渲染大盘。重点：**图像节点直接以圆形高保真缩略图悬浮渲染**，大幅提升视觉辨识度。
* **思想流光追踪 (Thought-Trace Highlighting)**：基于 WebSocket，当Agent在后台执行多跳寻路时，前端拓扑图实时点亮其思考游走过的路径。
* **微观 Wiki 抽屉 (Micro Wiki Panel)**：点击任意节点侧滑弹出的模块化编辑器（类似Notion），展示由Agent自动维护的长程传记及瀑布流图库。

### 4. 接口规范（核心：完全解耦的标准化持久层 SPI）

本设计强制执行**依赖倒置原则（DIP）**。核心领域模型**绝对禁止**包含任何 SQL、Cypher (Neo4j) 或 BSON (MongoDB) 的专有语法。所有数据交互必须通过以下四个独立定义的**端口（Ports / SPI）**进行。

不论后续人工提供何种物理存储（哪怕是退化到纯本地 TXT 文件），只需编写对应的 Adapter 适配器实现以下接口，即可被 IoC 容器无缝接管。

#### 4.1 全局二进制对象抽象接口 (`IBlobStoragePort`)
*职责：存储原始多媒体，不可分割、不可被LLM直接阅读的物理载体。*
```python
class IBlobStoragePort(ABC):
    @abstractmethod
    def upload_binary(self, file_stream: bytes, mime_type: str) -> str:
        """上传二进制文件，系统返回全局统一资源标识符 (如 blob://img_001)"""
        pass
        
    @abstractmethod
    def get_presigned_url(self, uri: str, expires_in_sec: int) -> str:
        """为前端渲染或VLM读取生成有时效的安全直链"""
        pass
```

#### 4.2 逻辑拓扑图抽象接口 (`IGraphStoragePort`)
*职责：承载微观、离散、结构化的实体关系骨架。*
```python
class IGraphStoragePort(ABC):
    @abstractmethod
    def upsert_node(self, node_uuid: str, label: str, properties: dict) -> bool:
        """幂等地创建或更新实体节点/图像节点"""
        pass
        
    @abstractmethod
    def upsert_edge(self, src_uuid: str, tgt_uuid: str, relation: str, weight: float) -> bool:
        """创建或更新有向连线"""
        pass
        
    @abstractmethod
    def traverse_subgraph(self, start_uuid: str, max_depth: int, filters: dict) -> list[GraphPathDTO]:
        """核心路由引擎：基于标准化参数执行多跳广度/深度遍历，屏蔽底层的Cypher或递归SQL实现"""
        pass
        
    @abstractmethod
    def human_override_delete_edge(self, src_uuid: str, tgt_uuid: str) -> bool:
        """人工逆向干预专用的硬删除接口"""
        pass
```

#### 4.3 维基长文叙事抽象接口 (`IDocumentStoragePort`)
*职责：承载宏观、连续、富含情感语境与细节的 Markdown长文。*
```python
class IDocumentStoragePort(ABC):
    @abstractmethod
    def save_wiki(self, entity_uuid: str, content_markdown: str, version_hash: str) -> bool:
        """覆写保存完整Wiki页，内部需处理乐观并发控制机制(OCC)"""
        pass
        
    @abstractmethod
    def get_wiki(self, entity_uuid: str, at_timestamp: float = None) -> DocumentDTO:
        """获取实体的维基正文。支持传入时间戳获取历史记忆快照"""
        pass
```

#### 4.4 跨模态统一语义抽象接口 (`IVectorStoragePort`)
*职责：承载将图像与文本降维对齐后的高维数学空间，实现模糊物理寻址。*
```python
class IVectorStoragePort(ABC):
    @abstractmethod
    def upsert_embedding(self, reference_uuid: str, vector_array: list[float], metadata: dict) -> bool:
        """绑定UUID与特征向量，支持动态更新"""
        pass
        
    @abstractmethod
    def similarity_search(self, query_vector: list[float], top_k: int, threshold: float) -> list[str]:
        """近似最近邻（ANN）检索，返回满足阈值的相关 UUID 列表"""
        pass
```

### 5. 跨模态统一数据模型设计 (Data Model)

在上述四分天下的存储介质中，由一个全局唯一的**超级指针（UUID）**贯穿融合。系统中流转的统一聚合根称为**记忆胶囊 (Memory Capsule)**：

**以一张“合影”的记忆摄入为例，底层数据的物理分布映射：**
1.  **Blob层**: 存入图片，返回 `blob://img_888`
2.  **Graph层**: 产生 `Node(uuid="img_888", type="IMAGE")` 以及连线 `(Person:张三)-[APPEARS_IN]->(Image:img_888)`
3.  **Doc层**: 更新张三对应的JSON文档：`{"doc_id": "张三_uuid", "markdown": "## 参会记录\n今天参加了会议。![会议现场](blob://img_888)"}`
4.  **Vector层**: 存放 `img_888` 的 CLIP 视觉向量，以及“参会记录”的文本Embedding，指向同一个语义簇。

### 6. 非功能需求约束 (NFRs)

#### 6.1 性能与系统吞吐量
*   **首字响应延迟 (TTFT)**：在 System 1 极速工作记忆流中，从接收请求到缓存特征提取完成并打入大模型，内部耗时需 `< 150ms`。
*   **异步沉淀吞吐量**：依托 Kafka + Memory Worker 集群的水平伸缩，支持跨模态图文反思的最终一致性同步延迟峰值控制在 `120秒` 以内。
*   **可视化渲染性能**：2.5D星图需应用 WebGL 视锥体剔除（Frustum Culling）技术，确保在普通办公电脑浏览器中流畅承载 `15,000+` 节点的高频交互，帧率持续 `> 40 FPS`。

#### 6.2 扩展性与热拔插基准
*   **零代码侵入替换**：业务层必须做到 100% 存储无感。从单机版（FAISS + SQLite图模拟 + 本地JSON）迁移至云原生企业版（Milvus + Neo4j集群 + MongoDB + AWS S3）时，**只需修改配置文件（Config/DI Registry），核心代码层变更量必须严格为 0**。

#### 6.3 安全隔离与数据治理
*   **多租户/多Agent强制隔离**：所有存储 Adapter 的底层实现必须拦截并校验 `Tenant_ID`。即便在图谱游走时，也严禁算法越过当前租户边界，杜绝数据越权。
*   **数据彻底遗忘权**：执行节点删除时，必须支持“跨介质级联擦除”，物理销毁该UUID对应的所有图连线、文档切片及源生图片。

### 7. 实施路径及验收标准

#### 7.1 分阶段实施路径 (Roadmap)
*   **Phase 1：内核基建与防腐验证 (M1)**
    *   确立四大 `Port` 接口防腐层契约。编写基于“纯内存+本地文件”的轻量级 Adapter。走通文本维基与关系图谱的双轨协同写入逻辑。
*   **Phase 2：多模态升维与分布式装配 (M2)**
    *   接入 Kafka 消息队列，部署异步 Saga 状态机。引入视觉模型（VLM）打通 `Image-as-a-Node` 全链路。研发并集成主流真实数据库（如 MySQL, Neo4j, MinIO）的 Adapter 驱动。
*   **Phase 3：可视化大屏落地与调优 (M3)**
    *   开发 3D WebGL 交互大屏与富文本抽屉。实现时空漫游滑块功能与“逆向删改API”的人机对抗测试。进行抗本体漂移算法压测。

#### 7.2 核心验收标准 (UAT Criteria)
1.  **架构极致解耦验收（断崖式替换测试）**：在集成测试环境中，临时将依赖注入配置文件中的图谱驱动从 `Neo4jAdapter` 切换为团队基于 MySQL 手写的 `RdbmsGraphAdapter`。重启服务后，系统的记忆读写、关联查询自动化测试通过率必须保持 100%，不发生任何崩溃。
2.  **多跳防幻觉与图文溯源验收**：导入包含10个人物、5个时间线段与8张工作照片的复杂记忆沙盒。向Agent提问跨度至少 3 跳的隐蔽逻辑（例如：“找出上个月与我滑雪照片里那个戴帽子男人的现任领导”）。Agent 需在 `< 3秒` 内准确回答，幻觉率为 0%，并能直接提取对应的合影照片 URI 作为证据链供 UI 渲染。
3.  **UI双向干预即时生效验收**：用户在 3D 星图界面上，用鼠标手动剪断实体A与实体B之间的业务连线。操作后系统四层物理库在 1 秒内同步刷新完毕；立刻通过对话框发起关于A和B关系的问题，Agent 必须表现出“物理级遗忘”该段关联的受控认知状态。
